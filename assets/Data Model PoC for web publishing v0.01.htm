1      Overview

The Detailed Conceptual Model (DCM) is a series of documents showing entities, their relationships and relevant attributes of both for each model diagram. The entities and relationships are built on the High-Level Conceptual Model (H-LCM: Data Architecture Requirements Appendix A.doc), with inputs from:

1.1   EMu

This existing CMS solution at The Museum. We have taken a full data dictionary of the database and combined this with field usage statistics, to update the model and provide mappings from EMu to this DCLM. This helps us ensure we have full coverage of the existing CMSs data requirements. The EMu data dictionary it split into modules, and this model will show which modules and field apply to which entities and attributes. The RECODE EMu DataDictionary Mappings list is the reference, showing the detailed mappings from EMu to the data model.

1.2   Functional Requirements

Information based around stakeholder definitions of what the new CMS will need to provide. These documents are available to bidders and the DCM addresses each the User Story and Data Element and defines a modelling solution to support them. The DCM shows the outcome of this process, with the mappings in a separate document.  

1.3   External schemas and standards

The DCM will put its objects into context, showing where they reference external standards (e.g. SPECTRUM from the Collections Trust) and schemas (e.g. Darwin Code from the Biodiversity Information Standards). The DCM shows where its objects and attributes are analogous to the external ones and explains any differences and complexities. 

1.4   Results 

These sources have led to: 

Additions and changes to the entities and relationships from the H-LCM, as the requirements are more complex 

The additions of attributes and attribute groups to the entities and relationships 

The aim is to give readers a full representation of The Museums interpretation of its collection data model, showing how it links to the relevant concepts from a variety of sources. This should give suppliers enough information to assess how the Museums data model could be implemented in their proposed solution and be the starting point for detailed discussions in the latter stages of the procurement process. 

2      Location

2.1   Location Summary

The location describes where objects within the collection are physically located. This may encompass:

         storage locations (including freezers)

         display locations

         offices and labs

         external locations (e.g. for touring exhibitions)

2.2   Related EMu Concepts

This diagram addresses the details of the elocations and emovements modules:

2.2.1   elocations

This Records storage and display locations within an institution, e.g. building, floor, room, cabinet, shelf, box. In addition to recording the location hierarchy, it captures physical attributes of that location e.g. dimensions.  X-Y-Z co-ordinates are used for spatial analysis of pest trap data using the mapper interface. All these properties are addressed here. Move requests and location history are also handled in this module and addressed in this diagram.

More work is needed for location condition checking and reporting requirements.

2.2.2   emovements

This records movements of collection items into and out of an institution. The core data for this module is addressed here, but some fields may be addressed within processes that specifically manage the processes that instigate moves into an out of the Museum (e.g. Loans).

2.2.3  Data Cleansing Areas

This section highlights potential areas that may need to be considered when mapping and cleansing the existing data. These are limited to points noticed in initial explorations of the EMu data and aren't intended to be exhaustive.

         Locations should be a strict hierarchy, and duplicates should be removed

         Indexing Details there are Collection Names (e.g. Crustacea) that seem to be above taxonomic Order, so appear to break the indexing details hierarchy. And there are many that appear to have no child locations. We also need to understand how the indexing is used and how it correlates to the levels physical location hierarchy.

         Location Type should be rationalised

2.5   Data Model

Location is where a CollectionObject is stored. This represents a hierarchy from site and building level down to granular containers such as a drawer or box.

There are multiple ways that Locations and CollectionObjects can be linked, and these are modelled using Type and Status in the ObjectLocation and LocationLocation entities. This allows different types of relationships to be described and to show movement histories.

2.5.1  Location

Name

Location

Definition

A Location is a position in a hierarchy of places where an object may be situated at a given time.

Scope

A Location may contain data about its type, and a link to a parent Location.

Context

A Location can be the normal location of a CollectionObject and, in addition to storage units, can include display cases, galleries, and laboratories; a location type attribute will be included to reflect these. Locations can be moveable (e.g. drawers, boxes and other containers), they still fit in this model so long as the parent is changed when one is moved.

A history of all locations at which objects are, or have been, located is retained for all objects, as well as locations reserved for objects that may be temporarily elsewhere (e.g. in an exhibition).

The building level Location should have an Address.

There will need to be access controls to the data as some locations are sensitive.

External References

CIDOC:

              This is a synonym of E53 Place.

              P87 is identified by links a place to a place appellation, and an address is a subtype of this.

Notes

we might need to have location values of "not in place" and "on loan" to show what we know about a location when it cannot be accurately defined

2.5.6.1.3      Role

Name

Role

Mandatory

Y

Type

CV

Definition

The role an Agent plays in the Location's hierarchy and changes to that.

Scope

 

Context

This might need to be informed by data on the CollectionObject if there are restrictions on its movement.

Examples

         Authoriser: the Person that Authorised a move.

         MovedBy: the Person that moves a CollectionObject to a different Location.

         Notify: Anyone who needs to be aware of the move.

         Receiver/Dispatcher

         Carrier
